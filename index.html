Общая архитектура
Как компоненты системы связаны между собой
Архитектура: Monolith + отдельные сервисы для чата и уведомлений. На этапе MVP нет смысла строить микросервисы — это дорого и медленно. Используем хорошо структурированный монолит с чёткими модулями, который легко разбить позже.

Аутентификация — JWT Flow
Схема: SMS OTP (через Eskiz UZ) → верификация → Access Token (15 мин) + Refresh Token (30 дней) в httpOnly cookie. Access Token хранится в памяти (не localStorage!). При каждом запросе — проверка в Redis blacklist (для логаута). Refresh Token rotation — при обновлении старый инвалидируется.
1. SMS OTP

Пользователь вводит телефон → сервер отправляет 6-значный код через Eskiz API → код хранится в Redis 5 минут → max 3 попытки.

2. Token Pair

При успешном OTP — сервер выдаёт: access_token (JWT, 15 мин) + refresh_token (UUID, 30 дней, httpOnly cookie).

3. Refresh

При истечении access — клиент делает POST /auth/refresh. Сервер проверяет refresh, выдаёт новую пару. Старый refresh инвалидируется.

07
Real-time / Чат
Socket.io · Redis Adapter · Горизонтальное масштабирование
Архитектура чата: WebSocket через Socket.io. Каждая conversation — отдельная комната (room). Redis Adapter позволяет горизонтально масштабировать WebSocket серверы — сообщения публикуются в Redis Pub/Sub и доставляются всем репликам.
Socket Events (клиент → сервер)
  • join_conversation — Подключиться к комнате conversation_id
  • send_message — Отправить сообщение: {text, conversation_id, type}
  • typing_start — Индикатор "печатает..."
  • typing_stop — Остановить индикатор
  • mark_read — Прочитать сообщения до message_id
Socket Events (сервер → клиент)
  • new_message — Новое сообщение в комнате с полными данными
  • message_read — Статус прочтения обновлён
  • user_typing — Другой участник печатает
  • user_online / user_offline — Статус онлайн участника
  • notification — Push-уведомление о новом сообщении

Стратегия индексации
  • GIST-индекс на listings.location (PostGIS) — быстрый геопоиск
  • GIN-индекс на listings.title + description — full-text search
  • BTREE-индексы на listings.price, district_id, status, created_at
  • Составной индекс (user_id, status) на listings — мои объявления
  • BTREE на messages.conversation_id + created_at — история чата
  • Partial index на listings WHERE status='active' — только активные
Партиционирование: Таблица messages делится по месяцам (PARTITION BY RANGE created_at). При 150K пользователях сообщений будет миллионы — партиции критичны для производительности.

Безопасность
OWASP Top 10 · Data Protection · Anti-fraud
🔐
JWT + Refresh Token Rotation
Access: 15 мин в памяти. Refresh: httpOnly cookie, Secure, SameSite=Strict. При краже refresh — автоинвалидация семьи токенов.
🚦
Rate Limiting
OTP: 3 попытки / 5 мин. Регистрация: 5 / час с одного IP. API: 100 req/мин. Redis sliding window.
🛡️
Input Validation
Все входные данные валидируются через Zod. SQL-инъекции невозможны — Prisma ORM с параметризованными запросами.
🔒
HTTPS + HSTS
SSL через Cloudflare. HSTS header. Security headers через helmet.js: X-Frame-Options, CSP, CORS только whitelist.
🖼️
Фото-загрузки
Presigned R2 URLs — клиент загружает напрямую, минуя сервер. Валидация MIME-type. Сканирование на NSFW через AI API.
🔑
Secrets Management
Все секреты в .env файлах, не в коде. В продакшне — через GitHub Secrets. Ротация API ключей каждые 90 дней.
👁️
Fraud Detection
Один телефон — один аккаунт. Лимит на объявления без верификации. AI-модерация текстов на спам/фейки.
📋
Audit Log
Все admin-действия логируются с actor_id, IP, timestamp. Удаление данных — soft delete (GDPR-совместимость).
